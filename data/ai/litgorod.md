# Litgorod Context

Litgorod is another fiction platform used for publishing and managing author materials. For Digital Shelf, treat Litgorod as a platform where users usually ask for workflow acceleration, simplification of repeated entry, and clearer author-side navigation.

## Core areas that matter for requests

- author cabinet navigation;
- book and chapter management related forms;
- statuses, descriptions, and text-entry blocks;
- publication-related editing routines;
- promotional text and supporting materials where the exact interface may vary over time.

## Conservative operating assumption

For Litgorod, the AI should behave more carefully than for other platforms. Exact interface details may be less visible in public sources or may change over time. The AI must therefore build useful ТЗ from general author-workflow logic, not from invented UI specifics.

## Typical repetitive work an author may want to automate

- filling repeated text fields;
- moving faster between several author sections;
- preparing book materials in a more structured way;
- reducing copy-paste and manual form editing;
- grouping routine actions into one lightweight helper.

## Good wording patterns for AI-generated ТЗ

When the user writes about Litgorod, the AI should often translate the request into developer language such as:

- helper for faster work in the Litgorod author cabinet;
- tool for repeated заполнение форм и карточек;
- extension for quick navigation between книгами, главами и статусами;
- assistant for editing descriptions and publication materials;
- utility that reduces manual repeated actions in Litgorod workflows.

## Frequent user intentions the AI should recognize

- "нужно быстрее работать" -> navigation helper or routine-acceleration tool;
- "нужно меньше копировать руками" -> snippet/template/autofill helper;
- "надо проще оформлять книги" -> structured editing helper for repeated fields;
- "нужен помощник в кабинете" -> compact utility that surfaces repeated actions or information;
- "нужно удобнее следить за материалами" -> tracking or summary helper around statuses and editing work.

## Good scope examples for Litgorod requests

- helper for repeated filling of author-side forms;
- compact panel for faster switching between common author sections;
- tool for reusing recurring descriptions, notes, or metadata fragments;
- assistant that reduces copy-paste when working with several books or materials;
- lightweight utility for routine publication preparation.

## Bad scope patterns the AI should avoid

- claiming exact hidden sections or exact button names;
- promising deep automation of undocumented platform internals;
- overcomplicating a request that is really about reducing manual routine;
- inventing analytics, billing, or admin capabilities not mentioned by the user.

## Desired output behavior for Litgorod

When producing ТЗ for Litgorod:

- keep wording practical and neutral;
- emphasize simpler routine, faster entry, better navigation, and lower manual effort;
- use conservative phrases like "в кабинете автора" or "в форме книги";
- keep the result credible even when interface specifics are unknown.

## Constraints

- Stay conservative: Litgorod interface details may differ, so avoid inventing exact control names.
- Prefer neutral wording such as "в кабинете автора", "в форме книги", "в разделе управления публикацией", "при редактировании материалов".
- If the user is vague, frame the task as workflow simplification, not as a risky deep integration claim.

## Useful source direction

This context is intentionally conservative and is designed to help the AI write safe, practical ТЗ without pretending to know hidden or unstable interface details.
